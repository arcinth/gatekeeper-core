# Purpose

This document defines the core business entities of GateKeeper and the relationships between them.

The Domain Model represents the business language used throughout the system. It is intentionally independent of any programming language, database technology, or API implementation.

This document serves as the foundation for:

- Database Design
- REST API Design
- Backend Domain Objects
- Frontend Models
- Engineering Reports
- Future Platform Extensions

---

# Domain Philosophy

GateKeeper is designed around one central business process:

> A Pull Request is analyzed by multiple analysis engines to produce engineering findings, which are evaluated to generate a governance decision before software reaches production.

Everything in the domain exists to support this workflow.

---

# Core Domain Objects

## Organization

Represents an engineering organization using GateKeeper.

An Organization owns repositories, users, engineering policies, and governance settings.

### Responsibilities

- Own repositories
- Manage engineering policies
- Define governance rules
- Manage users and roles

---

## User

Represents an authenticated engineering user.

A User interacts with GateKeeper according to their assigned role.

### Responsibilities

- View Pull Request reports
- Review findings
- Configure repositories (depending on permissions)
- Manage engineering policies (depending on permissions)

---

## Role

Represents a collection of permissions assigned to users.

Roles define which governance operations a user may perform.

Examples include:

- Developer
- Technical Lead
- Engineering Manager
- Platform Engineer
- DevSecOps Engineer

---

## Repository

Represents a GitHub repository governed by GateKeeper.

Repositories act as the boundary between GitHub and the analysis platform.

A Repository contains many Pull Requests.

---

## Pull Request

Represents a code change submitted for review.

A Pull Request is the primary object analyzed by GateKeeper.

A Pull Request may undergo multiple analyses during its lifecycle.

Each new commit creates a new Analysis Run.

---

## Analysis Run

Represents one execution of the GateKeeper analysis pipeline.

An Analysis Run records the complete evaluation of a Pull Request at a specific point in time.

Every Analysis Run is immutable once completed.

### Responsibilities

- Execute analysis engines
- Collect findings
- Track execution status
- Produce a governance result

---

## Analysis Engine

Represents an engine capable of analyzing a Pull Request.

Every engine implements the same conceptual contract.

Current implementations include:

- Policy Engine
- Security Engine
- AI Review Engine

Future implementations may include:

- Architecture Engine
- Performance Engine
- Compliance Engine
- Secrets Engine

The orchestration pipeline depends only on the Analysis Engine abstraction.

---

## Finding

Represents an observation produced by an Analysis Engine.

Finding is the common parent concept for all engineering observations.

Current finding types include:

- Policy Finding
- Security Finding
- AI Finding

Every finding has:

- Source Engine
- Severity
- Description
- Recommendation
- Classification

Findings are either:

### Deterministic

Generated by deterministic engines.

Used by the Verdict Engine.

Examples:

- Policy violations
- Security violations

### Advisory

Generated by AI.

Visible to developers.

Never influence governance decisions.

---

## Verdict

Represents the governance decision for an Analysis Run.

A Verdict is produced exclusively from deterministic findings.

Possible outcomes include:

- Passed
- Failed

The Verdict Engine is the sole owner of this decision.

---

## Engineering Report

Represents the unified report presented to developers.

A report combines:

- Policy findings
- Security findings
- AI observations
- Final governance verdict

Reports provide a single engineering view of a Pull Request.

---

## Audit Log

Represents an immutable history of significant governance events.

Audit Logs support:

- Explainability
- Compliance
- Governance
- Traceability

Every important action performed by GateKeeper is recorded.

# Domain Principles

The GateKeeper domain follows these principles:

- Pull Requests are immutable inputs.
- Analysis Runs are immutable records.
- Findings are owned by Analysis Runs.
- AI findings are advisory only.
- Deterministic findings produce governance decisions.
- Every governance decision is auditable.
- Every Analysis Engine follows a common abstraction.
- Every future analysis capability should integrate without modifying existing orchestration logic.

# Ubiquitous Language

| Term | Definition |
|------|------------|
| Pull Request | A code change submitted for governance analysis |
| Analysis Run | One execution of the analysis pipeline |
| Analysis Engine | A component that analyzes Pull Requests |
| Finding | An observation produced by an Analysis Engine |
| Deterministic Finding | A finding that participates in governance decisions |
| Advisory Finding | An AI-generated recommendation |
| Verdict | Final governance decision |
| Engineering Report | Unified presentation of findings and verdict |
| Governance | Enforcement of organizational engineering standards |

---

# Guiding Principle

> Every Pull Request produces one or more Analysis Runs.

> Every Analysis Run produces Findings.

> Deterministic Findings produce the Verdict.

> Advisory Findings improve engineering quality without influencing governance decisions.
